## API Key Management
- Format: `sk_ed_{UUID}.{secret-token}` where UUID is key ID and secret is 64-char hex
- Storage: ID + SHA-256 hash of secret with pepper salt (constant-time comparison)
- Verification: Extract ID from token, lookup by ID, compare hashes
- Authorization:
  - All authenticated keys: proxy endpoints (chat, completions, RAG)
  - Admin keys only: management mutations (create, revoke, rotate)
  - Admin flag controlled via `is_admin` boolean column
- Endpoints:
  - `POST /v1/management/api-keys` - create new key (admin-only, optionally with `is_admin: true`)
  - `POST /v1/management/api-keys/{id}/revoke` - deactivate key (admin-only)
  - `POST /v1/management/api-keys/{id}/rotate` - replace key with grace period (admin-only, `grace_period_sec` controls old key expiration, 0=immediate)
  - `GET /v1/management/api-keys` - list keys (any authenticated key)
  - `GET /v1/management/usage` - usage summary (any authenticated key)
- Cache invalidation: immediate eviction from auth + rate limit caches on mutation

## Dependency Health Coverage
`/healthz` now reports structured health telemetry for:
- PostgreSQL
- Redis
- LiteLLM
- Elasticsearch

Response semantics:
- Overall status is `healthy`, `degraded`, or `unhealthy`
- Reachable but slow dependencies are marked `degraded`
- Unreachable dependencies are marked `unhealthy`
- Host telemetry includes disk usage, memory usage, and CPU utilization
- `HEALTH_DEGRADED_LATENCY_MS` controls the degraded latency threshold and defaults to `2000`
- `/livez` is the process-only readiness/liveness check and is preferred for startup waits
- `HEALTH_CRITICAL_DEPENDENCIES` controls which unreachable dependencies force overall `unhealthy`
